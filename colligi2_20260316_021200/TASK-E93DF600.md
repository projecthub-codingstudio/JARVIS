## 1. 문서 목적 및 의사결정 요약

본 문서는 Company의 Project로 추진하는 `JARVIS 프로젝트`에 대해, MacBook Pro 16 / Apple M1 Max / 64GB 메모리 환경에서 실제로 설계·구현 가능한 로컬 AI 비서의 실행 구조를 정의하는 기술 실행 문서다. 목적은 가능성 검토에 머무르지 않고, 어떤 범위를 먼저 구현해야 하며 무엇을 당장 하지 말아야 하는지까지 포함한 의사결정 기준을 제공하는 데 있다.

핵심 결론은 다음과 같다.

- 이 프로젝트는 `아이언맨형 범용 자율 비서`로 접근하면 실패 확률이 높지만, `로컬 퍼스트 개인 워크스페이스 에이전트`로 재정의하면 충분히 추진 가능하다.
- 성공의 본질은 모델 크기 경쟁이 아니라 `검색 품질`, `오케스트레이션`, `권한 신뢰`, `시스템 공존성`에 있다.
- M1 Max 64GB 환경에서 기본 상주 모델은 `14B급`이 현실적이며, `30B 내외 모델`은 상시가 아니라 심화 추론용 승격 계층으로 다루는 것이 타당하다.
- MVP는 `텍스트 중심`, `선택 폴더 기반`, `읽기 + 인용 + 초안 생성`에 한정해야 한다.
- 음성, 화면 인식, 접근성 기반 UI 자동화, 일반 셸 자율 실행은 초기 가치 검증 이전에 넣지 않는다.
- 이 문서의 승인 기준은 “돌아가는 데모”가 아니라 `5일 연속 실제 업무에서 하루 3회 이상 자발적으로 사용되는가`다.

## 2. 문제 재정의

초기 질문은 “개인 노트북 안에 JARVIS 같은 로컬 AI 비서를 만들 수 있는가”였지만, 실행 관점에서 더 정확한 질문은 다음과 같다.

`개인 노트북의 제약 안에서, 한국어 질의응답·로컬 검색·작업 초안 생성을 안정적으로 수행하고, 사용자의 문서·코드·작업 맥락을 근거와 함께 연결하는 로컬 퍼스트 워크스페이스 에이전트를 구축할 수 있는가.`

이 재정의 아래에서 먼저 코드로 입증해야 할 가정은 세 가지다.

1. 한국어 검색이 실제 개인 문서와 코드 코퍼스에서 충분한 정확도를 내는가
2. 런타임 오케스트레이터가 메모리 압박, 배터리, 발열, 백그라운드 인덱싱을 통제할 수 있는가
3. 최소 수직 슬라이스가 실제 업무에 재사용될 정도의 신뢰와 편의를 주는가

이 프로젝트는 챗봇 개발이 아니라 `개인 지식 기반 운영체계의 초석`을 만드는 작업에 가깝다. 따라서 언어 모델보다 `검색 인프라`, `권한 정책`, `실패 모드`, `운영 정책`이 더 중요하다.

## 3. 최종 제품 정의

JARVIS 프로젝트의 1차 제품 정의는 다음과 같다.

`한국어로 내 문서와 코드의 맥락을 이해하고, 근거를 인용하며, 실제 작업 전 초안을 제시하고, 승인된 범위 안에서만 제한적으로 도와주는 로컬 퍼스트 개인 AI 시스템`

이 정의에 따라 1차 제품은 아래를 포함한다.

- 선택 폴더 인덱싱
- 문서·코드 통합 검색
- 근거 인용형 질의응답
- 작업 초안 생성
- 대화 히스토리 및 작업 맥락 축적
- 승인형 제한 실행

이 정의에 따라 1차 제품은 아래를 제외한다.

- 무제한 파일 접근
- 접근성 권한 기반 범용 UI 조작
- 일반 셸 자율 실행
- 음성 상시 대기
- 화면 전체 문맥 상시 수집
- “알아서 다 해주는” 완전 자율 에이전트

## 4. 하드웨어 현실과 설계 함의

Apple M1 Max 64GB는 로컬 AI 실행에 매우 유리한 장비이지만, 설계 판단은 벤치마크가 아니라 실제 병행 사용 환경을 기준으로 내려야 한다.

### 4.1 하드웨어 특성
- 통합 메모리 64GB
- 메모리 대역폭 약 `200GB/s`
- CPU, GPU, 메모리 풀이 공유되는 Apple Silicon 구조
- 실제 사용 시 IDE, 브라우저, Docker, 디자인 툴, 메신저와 메모리 경쟁 발생

### 4.2 설계 함의
- “모델이 로드된다”와 “업무 중 상시 쓸 만하다”는 다르다.
- 30B 이상 계열을 기본값으로 두면 swap, memory pressure, 앱 전환 지연이 발생할 가능성이 높다.
- SSD swap 증가는 성능 저하뿐 아니라 장기적으로 쓰기량 증가 비용을 만든다.
- 배터리 모드에서는 추론과 인덱싱 정책이 별도로 필요하다.
- 따라서 이 프로젝트의 차별점은 대형 모델 탑재가 아니라 `상황별 제어 정책`이다.

## 5. 아키텍처 원칙

본 프로젝트의 핵심 원칙은 다음과 같다.

- `로컬 퍼스트`: 기본 처리는 로컬에서 수행하고, 외부 의존은 정책적으로 분리한다.
- `검색 우선`: 생성 품질보다 검색 품질을 먼저 확보한다.
- `근거 우선`: 사실성 응답은 반드시 출처를 동반한다.
- `최소 권한`: 기능보다 권한을 늦게 연다.
- `승인 기본`: 상태 변경 작업은 사용자 승인 전 실행하지 않는다.
- `공존성 우선`: 노트북은 AI 전용 장비가 아니므로 업무 앱과 함께 살아야 한다.
- `측정 기반`: 모든 주요 판단은 계측값과 실패 기준을 갖는다.
- `확장 전 검증`: 음성·화면·자동화는 MVP 가치가 입증된 뒤 확장한다.

## 7. 계층 아키텍처

최종 구조는 이해관계자 설명에 유리한 4계층 프레임을 사용하되, 내부 구현은 세분화한다.

### 7.1 상위 4계층
1. `인터페이스 계층`
   - CLI REPL
   - 메뉴바 UI
   - 승인/거부, 상태 표시, 인용 표시

2. `오케스트레이션 계층`
   - 의도 분류
   - 모델 라우팅
   - 권한 상태 관리
   - 안전 정책 적용
   - 성능 제어

3. `지식 계층`
   - 파일 수집
   - 청킹
   - FTS 인덱스
   - 벡터 인덱스
   - 대화·작업 메모리
   - 최신성 검증

4. `모델 런타임 계층`
   - 생성 모델
   - 임베딩 모델
   - 재순위화 모델
   - 추론 엔진

### 7.2 상세 구성요소
| 계층 | 구성요소 | 역할 |
|---|---|---|
| 인터페이스 | CLI, 메뉴바, 승인 패널 | 질의, 응답, 출처 표시, 실행 승인 |
| 오케스트레이션 | Governor, Planner, ToolRegistry | 모델 승격, 컨텍스트 제한, 툴 호출 통제 |

## 8. 데이터 흐름과 제어 흐름

### 8.1 질의응답 흐름
1. 사용자가 한국어 질의를 입력한다.
2. 오케스트레이터가 질의 유형을 판별한다.
3. Retriever가 FTS와 벡터 검색을 병행 실행한다.
4. Freshness Checker가 stale 결과 여부를 점검한다.
5. Context Assembler가 상위 근거와 대화 맥락을 조합한다.
6. Governor가 현재 메모리·배터리·발열 상태에 맞춰 모델 계층을 결정한다.
7. 생성 모델이 답변 초안을 생성한다.
8. Citation Enforcer가 근거 없는 문장을 감지해 보정한다.
9. 인터페이스가 답변과 파일 경로, 청크 근거, 신뢰도 배지를 표시한다.

### 8.2 승인형 작업 흐름
1. 사용자가 “정리해 줘”, “초안 만들어 줘”, “파일 이동해 줘” 같은 요청을 한다.
2. 시스템은 곧바로 실행하지 않고 `계획`과 `영향 범위`를 먼저 제시한다.
3. 사용자가 승인하면 제한된 툴만 실행한다.
4. 실행 결과는 로그와 되돌림 정보와 함께 저장한다.
5. 승인 없는 쓰기·삭제·외부 전송은 금지한다.

## 9. 핵심 기술 선택과 선정 기준

### 9.1 생성 모델 전략
현재 시점에서 특정 모델을 최종 확정하지 않는다. 대신 아래 후보군을 Phase 0 벤치마크 대상으로 둔다.

- `Qwen3-14B` 계열
- `EXAONE Deep 7.8B` 계열
- `Kanana-2-30B-A3B` 계열 또는 동급 한국어 강점 모델

선정 기준은 다음과 같다.

- 한국어 설명 품질
- 코드 이해 및 수정 지시 이행력
- 인용 기반 답변 일관성
- TTFT와 tokens/sec
- 메모리 점유와 swap 유발 정도
- 장시간 세션 안정성

결정 원칙은 다음과 같다.

- 기본 모델은 `14B급 또는 그 이하에서 실사용 품질이 확보되는 모델`
- 상위 모델은 `명시적 승격 또는 복잡한 추론 상황에서만 사용`
- 30B 내외 모델이 한국어 품질에서 유의미한 우위를 보일 경우에도 기본 상주는 금지
- 모델 교체 비용을 줄이기 위해 런타임 인터페이스를 표준화한다

### 9.2 런타임
- `MLX`를 1순위로 채택한다.
- 이유는 Apple Silicon 최적화, 메모리 효율, 로컬 배포 단순성 때문이다.
- 단, 모델 호환성 이슈가 있으면 `llama.cpp`를 대체 경로로 둔다.
- 따라서 결정은 `MLX 우선, llama.cpp 호환 확보`다.

### 9.3 검색
- `SQLite FTS5 + 벡터DB + 하이브리드 검색 + RRF`를 채택한다.
- 이유는 코드 식별자, 파일명, 에러 메시지, 한국어 자연어 질의가 하나의 검색 방식으로 해결되지 않기 때문이다.
- 벡터 단독은 코드베이스와 식별자 검색에서 약하고, FTS 단독은 의미 유사성 검색에서 약하다.
- reranker는 Phase 1에서 필수로 넣지 않고, 검색 정확도 미달 시 Phase 2에서 추가한다.

### 9.4 한국어 형태소 처리
후보는 다음 두 축으로 본다.

- `Kiwi`
  - 장점: Python 통합이 쉽고 설치 부담이 낮다.
  - 단점: 일부 도메인 토큰 처리에서 사용자 사전 조정 필요.

- `MeCab-ko`
  - 장점: 전통적으로 안정적이고 형태소 분석 품질이 검증됐다.
  - 단점: macOS 설치·배포 복잡도가 높고 개발환경 편차가 생길 수 있다.

초기 방침은 다음과 같다.

- MVP 기본값은 `Kiwi 우선`
- 한국어 검색 정확도가 목표 미달이면 `MeCab-ko` 경로를 활성화
- 코드 심볼, 파일명, 에러 문자열은 형태소 분석과 별도로 `원문 토큰 보존` 인덱스를 유지

## 10. 권한 및 보안 설계

이 프로젝트의 보안은 선언이 아니라 제품 구조여야 한다. macOS TCC는 불편한 장애물이 아니라 사용자가 신뢰를 판단하는 핵심 UX다.

### 10.1 권한 사다리
1. `권한 0 단계`
   - 수동 입력 텍스트만 처리
   - 인덱싱 없음
   - 가치 체험용 데모

2. `선택 폴더 단계`
   - 사용자가 지정한 폴더만 읽기
   - 문서·코드 검색 가능
   - MVP 기본 단계

3. `제한적 쓰기 단계`
   - 초안 파일 생성, 임시 폴더 출력
   - 반드시 승인 필요

4. `고급 자동화 단계`
   - Apple Events 또는 App Intents 기반 제한 연동
   - 별도 설정과 위험 고지 필요

5. `접근성 단계`
   - 범용 UI 자동화
   - MVP 및 초기 상용화 범위에서 제외

### 10.2 안전 정책
- 사실성 응답은 파일 경로와 청크 근거를 표시한다.
- 근거 없는 단정은 신뢰도 경고를 표시한다.
- 삭제, 이동, 실행, 외부 전송은 기본 금지다.
- 승인 토큰 없는 상태 변경 작업은 차단한다.
- 비정상 루프, 반복 실패, 과도한 파일 접근 시 Hard Kill이 즉시 동작해야 한다.

### 10.3 Hard Kill 및 실패 차단
- 모델 응답이 파괴적 명령으로 분류되면 실행 경로 차단
- 연속 오류 임계치 초과 시 툴 호출 정지
- 인덱서가 과도한 CPU 또는 디스크 쓰기를 유발하면 백오프
- macOS 업데이트 후 자동화 회귀 감지 시 자동화 기능 전체 비활성화

## 11. 데이터 생명주기와 최신성 관리

로컬 RAG 시스템에서 가장 위험한 실패는 “그럴듯하지만 오래된 근거를 사용하는 오답”이다. 따라서 인덱싱은 수집보다 `정화와 최신성`이 중요하다.

### 11.1 수집 원칙
- 사용자가 지정한 폴더만 인덱싱
- 파일 유형별 파서 분리
- 문서, Markdown, 코드, 로그, 노트 파일 우선
- 바이너리 대용량 파일은 기본 제외

### 11.2 최신성 정책
- FSEvents 기반 증분 인덱싱
- 파일 수정 시 메타데이터 변경 우선 반영
- 임베딩 재생성은 지연 큐로 처리
- 최근 수정 파일은 검색 점수에 freshness 보정 적용
- 삭제 파일은 tombstone 처리 후 색인 정리

### 11.3 저장소 비용 통제
- 대용량 저장소는 파일 크기와 변경 빈도 기준으로 제외 규칙 적용
- 임베딩 재생성 빈도 상한 설정
- 대화 로그는 전부 영구 보관하지 않고 요약 계층과 원본 계층 분리
- SSD 쓰기량과 인덱스 증가량을 주간 단위로 기록

## 12. 성능·배터리·발열 운영 정책

Gemini 초안이 제기한 에너지 관점은 반드시 포함되어야 한다. 이 프로젝트에서 팬 소음, 배터리 저하, 발열은 단순 성능 문제가 아니라 채택률 문제다.

### 12.1 Governor 정책
Governor는 아래 지표를 지속 관측한다.

- memory pressure
- swap 사용량
- CPU/GPU 사용률
- thermal state
- 전원 연결 여부
- 배터리 잔량
- 인덱싱 큐 길이

### 12.2 모드별 정책
- `전원 연결 + 유휴`
  - 상위 모델 승격 허용
  - 임베딩 재생성 적극 수행

- `전원 연결 + 업무 병행`
  - 기본 14B 유지
  - 인덱싱 저우선순위 실행

- `배터리 모드`
  - 상위 모델 승격 원칙적 금지
  - 인덱싱 일시 중지 또는 축소
  - 긴 컨텍스트 응답 제한

- `고발열/고압박 모드`
  - 컨텍스트 크기 축소
  - 모델 언로드
  - 답변 품질보다 응답성과 시스템 안정성 우선

### 12.3 권장 임계 정책
정확한 수치는 초기 측정 후 보정하되, 출발 정책은 다음과 같다.

- swap이 의미 있게 발생하면 상위 모델 즉시 금지
- thermal state 상승 시 인덱싱 백오프
- 배터리 30% 이하에서는 긴 추론 차단
- 첫 토큰 지연이 기준을 넘으면 컨텍스트와 검색 개수 축소

## 14. 실패 기준, 중단 기준, Plan B

시니어 레벨 문서라면 “계속할 이유”뿐 아니라 “멈출 기준”도 있어야 한다.

### 14.1 실패 기준
- 실제 코퍼스 50개 질의에서 Top-5 정확도 80% 미달
- 사실성 답변에서 인용 누락률 5% 초과
- 업무 병행 상태에서 memory pressure가 반복적으로 악화
- 5일 사용 실험에서 자발 사용 기준 미달
- 배터리 모드 불만이 반복적으로 제기됨

### 14.2 중단 기준
- 검색 신뢰 확보 없이 자동화 기능 확장 요구가 우세할 때
- 최소 권한 구조를 유지할 수 없을 때
- 모델 품질 향상을 위해 과도한 외부 서비스 의존이 필요할 때
- 구현 복잡도가 사용자 가치보다 빠르게 커질 때

### 14.3 Plan B
- `14B 품질 미달` 시:
  - 후보 2, 3 재평가
  - 한국어 특화 모델 라우팅 검토
  - 검색 품질 개선 후 재측정

- `FTS+벡터 하이브리드 성과 미달` 시:
  - chunk 전략 조정
  - reranker 도입
  - 코퍼스별 인덱싱 규칙 분리

- `MLX 호환성 이슈` 시:
  - llama.cpp 경로로 전환
  - 상위 계층 인터페이스는 유지

- `자동화 안정성 미달` 시:
  - 실행 기능 축소
  - 초안·추천 모드로 회귀

## 19. Colligi2 기반 문서 생성 방식

본 문서는 Colligi2의 집단 AI 문서화 프로세스를 통해 정리되었다. 생성 방식은 단일 모델 요약이 아니라 `다중 AI 충돌-수렴형 검토`다.

적용된 단계는 다음과 같다.

1. 원 요청의 표면 질문을 실행 가능한 문제로 재정의
2. 각 AI가 아키텍처, 성능, 보안, 한국어 처리, 로컬 퍼스트 관점에서 독립 초안 작성
3. 초안 간 충돌 지점을 비교하고, 과장·불일치·누락 항목을 분리
4. 리뷰 단계에서 실행 가능성, 리스크, 검증 게이트, 실패 기준을 중심으로 재평가
5. 최종 단계에서 평균적 절충이 아니라 `근거가 더 강한 결정`과 `남겨야 할 이견`을 유지한 채 문서 통합

따라서 이 문서는 단순 요약본이 아니라, 상충하는 제안 중에서 Project의 실행 가능성을 높이는 선택만 남긴 Company용 의사결정 문서다.

## 20. 최종 권고

Company의 Project로서 JARVIS 프로젝트는 추진 가치가 충분하다. 다만 출발점은 “로컬에서 큰 모델을 돌릴 수 있는가”가 아니라 “내 문서와 코드에 대해 신뢰 가능한 근거를 제시하며, 실제 업무 중에도 방해되지 않는가”여야 한다.

즉시 채택할 권고는 다음과 같다.

- `14B급 기본 모델 + 상위 모델 승격 구조`
- `MLX 우선, llama.cpp 호환 확보`
- `SQLite FTS5 + 벡터DB + 하이브리드 검색`
- `선택 폴더 기반 인덱싱`
- `인용 의무화와 승인형 실행`
- `배터리·발열·메모리 상태를 반영하는 Governor`

즉시 금지할 권고는 다음과 같다.

- Full Disk Access 기본 요구
- 접근성 권한 기반 범용 UI 자동화
- 일반 셸 자율 실행
- 음성/화면 기능 선탑재
- 검색 신뢰 확보 전 자동화 범위 확대

이 프로젝트의 올바른 1차 목표는 다음 한 문장으로 요약된다.

`한국어로 내 코드와 문서를 근거와 함께 설명하고, 필요한 작업은 실행 전에 초안과 영향 범위를 먼저 제시하는 로컬 퍼스트 워크스페이스 에이전트`

이 목표를 충족하면 JARVIS는 확장 가능한 시스템이 된다. 충족하지 못하면 그 어떤 화려한 기능도 장기적으로 채택되지 않는다.
